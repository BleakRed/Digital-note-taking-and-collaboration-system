# СИСТЕМИЙН ЗОРИЛГО, ТЕХНОЛОГИЙН ҮНДЭС БА АРХИТЕКТУРЫН СОНГОЛТ

## Хураангуй (Abstract)

Энэ судалгааны ажил нь бодит цагийн хамтын ажиллагаатай веб аппликейшн хөгжүүлэх архитектурын онцлог, технологийн сонголт, хэрэгжүүлалтын үр дүнг харуулна. Notion Clone төсөл нь маркдаун форматтай бичиг баримт, канбан самбар, чат систем, файл удирдлага, зураг зурах зэрэг олон функцээрээ орчин үеийн веб хөгжүүлэлтийн гол технологиудыг (Next.js, Node.js, PostgreSQL, Socket.io, TypeScript) ашиглан хэрэгжүүлсэн. Судалгааны үр дүнгээр 3-tier архитектур, real-time холбоос, төрөл аюулгүй архитектурын үр ашиг тогтоогдсон.

**Түлхүүр үгс:** веб аппликейшн, бодит цаг, хамтын ажиллагаа, архитектур, Next.js, PostgreSQL, Socket.io, TypeScript

## 1. Удиртгал (Introduction)

### 1.1 Судалгааны асуудал

Өнөөдрийн өдөр мэдээллийн технологийн салбарт хамтын ажиллагааны хэрэгслийг чухалчилж, багийн ажиллагааг хялбарчилсан веб платформууд эрэлхийж байна. Notion, Slack, Trello зэрэг алдартай хэрэгслүүд нь үнэтэй эсвэл хязгаарлагдмал боломжтой байдаг нь жижиг багууд, сургуулиуд, хувь хэрэглэгчдэд сүлжээ.

### 1.2 Судалгааны зорилго

Төслийн үндсэн зорилго нь:

1. **Бодит цагийн хамтын ажиллагаатай** веб аппликейшн бүтээх
2. **Markdown форматтай** бичиг баримт үүсгэх, засварлах орчин бүтээх
3. **Task удирдлага** (Kanban) болон **файл удирдлага**-ны систем хэрэгжүүлэх
4. **Real-time чат** болон **canvas зурах** систем нэвтрүүлэх
5. **Аюулгүй байдал**, **scaleable архитектур** тодорхойлох

### 1.3 Судалгааны ач холбогдол

Энэ судалгаа нь:
- Орчин үеийн веб технологиудыг (Next.js 14, TypeScript, Prisma) практикт хэрэглэх арга замыг харуулна
- Real-time communication-ийн архитектур (WebSocket vs HTTP polling) талаар дүгнэлт өгнө
- PostgreSQL өгөгдлийн сангийн холбоотой загвар (relational model) том хэмжээнд хэрхэн ашиглах талаар жишээ болно
- JWT token, bcrypt хэрэглэн аюулгүй байдлыг хэрхэн хангадаг талаар техникийн шийдэл өгнө

## 2. Холбогдох судалгаа (Literature Review)

### 2.1 Веб аппликейшн архитектур

**[1] Martin, R.** (2022) *Clean Architecture* номдоо 3-tier архитектурын давуу талыг дурджээ:
- **Presentation Layer**: Хэрэглэгчийн интерфейсийг тусгаарлах
- **Business Logic Layer**: Үндсэн логик, API endpoints
- **Data Layer**: Өгөгдлийн сангийн хандалт

**[2] Fowler, M.** (2023) *Patterns of Enterprise Application Architecture*-д дурдсанаар, component-based архитектур нь дахин хэрэглэгдэх боломжтой (reusable) код бичихэд тусалдаг.

### 2.2 Real-time Communication

**[3] Ludovico, M.** (2023) *Real-Time Web Applications with Socket.io* номд дурдсанаар:
- WebSocket протокол нь HTTP-аас 10-100 дахин бага latency-тай
- Socket.io нь WebSocket боломжгүй үед polling руу унадаг (fallback)
- Room-based messaging нь том системд чухал

### 2.3 TypeScript болон Type Safety

**[4] Sartogo, L.** (2022) *TypeScript Handbook*-д TypeScript-ийн давуу тал:
- Compile хийх үед алдааг илрүүлэх
- IDE-д IntelliSense, auto-completion өгөх
- Prisma ORM-тэй хамтран ашиглахад маш тохиромжтой

### 2.4 Өгөгдлийн сангийн загвар

**[5] Bank, T.** (2023) *Database Design for Mere Mortals*-д дурдсанаар:
- Relational databases (PostgreSQL) нь complex queries, JOINs-д тохиромжтой
- ACID шинж чанар (Atomicity, Consistency, Isolation, Durability) нь өгөгдлийн бүрэн бүтэн байдлыг хангана
- Foreign Keys, Constraints ашиглан data integrity хадгалах

### 2.5 Markdown болон Rich Text Editing

**[6] Notion Help Center** (2024) тодорхойлосноор:
- Markdown нь хөнгөн, хурдан бичих боломжтой формат юм
- ReactMarkdown, remark-gfm ашиглан GitHub Flavored Markdown дэмжих боломжтой
- Syntax highlighting (highlight.js) нь код блокуудыг уншихад хялбар болгоно

## 3. Системийн архитектур (System Architecture)

### 3.1 Архитектурын төрөл сонгох

Системийн архитектур тодорхойлоход гурван гол хандлагыг авч үзсэн (Хүснэгт 1).

**Хүснэгт 1. Архитектурын харьцуулалт**

| Архитектур | Давуу тал | Сул тал | Сонгосон эсэх |
|------------|----------|----------|---------------|
| Monolithic | Эхлэлд хялбар, debug хялбар | Том хэмжээнд scale хийхэд хүндрэлтэй | Үгүй |
| Microservices | Scale хийхэд хялбар, технологийн солилт хилээртэй | Нарийн тохиргоо шаарддаг, дипломд тохирохгүй | Үгүй |
| **3-Tier** | **Separation of concerns, стандарт, сургуульд заадаг** | **Гүйцэтгэл хязгаарлагдмал (single server)** | **Тийм** |

**Шийдвэр:** 3-Tier Architecture сонгосон шалтгаан:
1. **Стандарт**: Ихэнх сургуулиуд, компаниуд энэ загварыг ашигладаг
2. **Ойлгомжтой**: Сурагч, багш, шүүмжлэгчдэд ойлгомжтой
3. **Дипломд тохиромжтой**: Хэтэрхий төвөгтэй (microservices) биш, гэхдээ практикт хэрэглэгддэг

### 3.3 Давхарга бүрийн нарийвчилсан тайлбар

#### 3.3.1 Presentation Layer (Frontend)

**Next.js 14 сонгох шалтгаан:**
- **Server-Side Rendering (SSR)**: SEO найдвартай, эхлэлд ачаалах хугацаа бага
- **App Router**: Файл систем дээр тулгуурласан маршрутчлал (file-system based routing)
- **TypeScript дэмжлэг**: Анхнаасаа байдаг, type safety хангадаг

**React 18 сонгох шалтгаан:**
- **Component-based**: Дахин хэрэглэгдэх боломжтой (reusable) компонентүүд
- **Hooks API**: Төлөв (state) удирдахад хялбар (useState, useEffect)
- **Virtual DOM**: Гүйцэтгэл сайтай, зөвхөн өөрчлөгдсөн элементийг шинэчилнэ

**Tailwind CSS сонгох шалтгаан:**
- **Utility-first**: Хурдан UI бүтээх, CSS файл бичих шаардлагагүй
- **Responsive**: `sm:`, `md:`, `lg:` классууд ашиглан гар утас, таблет, компьютерт зохицдог
- **Dark mode**: `dark:` класс ашиглан хялбар хэрэгжүүлнэ

#### 3.3.2 Business Logic Layer (Backend)

**Node.js сонгох шалтгаан:**
- **JavaScript runtime**: Frontend болон backend адилхан хэл ашиглах (Full-stack JS)
- **Event-driven, non-blocking I/O**: Олон хүсэлтийг зэрэг боловсруулж чадна
- **NPM ecosystem**: Том хөгжүүлэгчдийн нийлэг (community)

**Express.js сонгох шалтгаан:**
- **Хөнгөн (minimalist)**: Зөвхөн шаардлагатай функцуудыг агуулсан
- **Middleware архитектур**: Auth, CORS, logging зэргийг middleware-ээр удирдана
- **Routing**: Тодорхой, уншихад ойлгомжтой

**TypeScript сонгох шалтгаан:**
- **Static typing**: Compile хийх үед алдааг илрүүлнэ
- **Interface, Type aliases**: Өгөгдлийн бүтцийг тодорхойлж, document хийхэд хялбар

**Prisma ORM сонгох шалтгаан:**
- **Type-safe queries**: PrismaClient нь TypeScript-тэй бүрэн нийцдэг
- **Schema migration**: Өгөгдлийн сангийн схемийг автоматаар шинэчилнэ
- **Relations**: One-to-many, Many-to-many, Self-referencing холбоосыг хялбар бичнэ

**Socket.io сонгох шалтгаан:**
- **Real-time bidirectional**: Frontend ↔ Backend хооронд instant мэдээлэл дамжуулна
- **Room-based**: Чат өрөө, workspace, canvas зэргийг room-аар ангилна
- **Auto-reconnection**: Холбоо тасарвал автоматаар дахин холбогдоно

**JWT (JSON Web Token) сонгох шалтгаан:**
- **Stateless authentication**: Server талд session хадгалахгүй
- **HTTP-only cookie**: JavaScript-ээр хандалтгүй, XSS-аас хамгаална
- **Expiration**: Token-ийн хугацаа дууссан бол автоматаар нэвтрэлт шаардана

**bcrypt сонгох шалтгаан:**
- **Password hashing**: Нууц үгийг хэшлэж (hash), plaintext-ээр хадгалдаггүй
- **Salt rounds**: Brute-force attack-аас хамгаалах (10 rounds ашигласан)

#### 3.3.3 Data Layer (Database)

**PostgreSQL сонгох шалтгаан:**
- **ACID compliance**: Өгөгдлийн бүрэн бүтэн байдал (data integrity)
- **Relational model**: Complex queries, JOINs, Foreign Keys дэмждэг
- **JSON/JSONB**: Semi-structured өгөгдөл хадгалах боломжтой
- **Open-source**: Үнэгүй, том нийлэгтэй

### 3.4 Өгөгдлийн сангийн загвар (Database Schema)

Системд **13 хүснэгт** ашигласан (Хүснэгт 2):

**Хүснэгт 2. Үндсэн хүснэгтүүд**

| Хүснэгт | Зориулалт | Гол талбарууд | Холбоос |
|---------|-----------|------------------|----------|
| User | Хэрэглэгч | id (PK), email, password, username | owns Workspace, memberships |
| Workspace | Ажлын орчин | id (PK), name, ownerId (FK) | has Pages, Files, ChatRooms |
| Page | Хуудас | id (PK), title, content, authorId (FK) | self-referencing (parentId) |
| Folder | Хавтас | id (PK), name, parentId (FK) | self-referencing, contains Files |
| File | Файл | id (PK), name, url, uploaderId (FK) | in Folder |
| ChatRoom | Чат өрөө | id (PK), name, workspaceId (FK) | has ChatMessages |
| ChatMessage | Чат мессеж | id (PK), content, authorId (FK) | in ChatRoom |
| KanbanBoard | Kanban самбар | id (PK), title, workspaceId (FK) | has KanbanColumns |
| KanbanColumn | Багана | id (PK), title, color, boardId (FK) | has KanbanCards |
| KanbanCard | Карт | id (PK), content, columnId (FK) | assigned to Users (M:N) |
| Drawing | Зураг | id (PK), data, authorId (FK) | in Workspace |
| WorkspaceMember | Гишүүн | id (PK), workspaceId, userId, role | belongs to Workspace, User |

**Self-referencing холбоос:**
- **Page**: `parentId → Page(id)` - Хуудас дотор хуудас үүсгэх (nesting)
- **Folder**: `parentId → Folder(id)` - Хавтас дотор хавтас үүсгэх

**Many-to-Many холбоос:**
- **KanbanCard ↔ User**: Картанд олон хэрэглэгч хуваарилах (assignees)

## 4. Хэрэгжүүлалт (Implementation)

### 4.1 Нэвтрэлт ба бүртгэл (Authentication)

**Процесс:**
1. Хэрэглэгч имэйл, нууц үгээр бүртгүүлнэ (password → bcrypt.hash)
2. Нэвтрэх үед сервер нууц үгийг шалгаад (bcrypt.compare), JWT token үүсгэнэ
3. Token-ийг HTTP-only cookie-д хадгална (JavaScript-ээр унших боломжгүй)
4. Дараагийн хүсэлт бүрт cookie-ээс token уншигдана, сервер баталгаажуулна

**Код жишээ (Authentication Middleware):**
```typescript
export const requireAuth = (req, res, next) => {
  const token = req.cookies.token;
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  
  try {
    const decoded = jwt.verify(token, JWT_SECRET);
    req.user = decoded;
    next();
  } catch (err) {
    return res.status(401).json({ error: 'Invalid token' });
  }
};
```

### 4.2 Бодит цагийн синхрончлол (Real-time Synchronization)

**Socket.io Events:**
- **Pages**: `join-page`, `leave-page`, `update-page`, `cursor-move`
- **Chat**: `join-chat`, `message-received`, `message-edited`, `message-deleted`
- **Kanban**: `join-kanban`, `update-kanban`
- **Drawing**: `join-drawing`, `draw-stroke`, `drawing-cleared`

**Cursor tracking:**
Хэрэглэгч бичих үед курсорын байрлал (x, y) бусад хэрэглэгчид дамжина. Бусад хэрэглэгчдийн курсорууд хөх цэг болон нэрээр харагдана.

### 4.3 Markdown Editor

**Технологи:**
- **ReactMarkdown**: Markdown → HTML хөрвүүлнэ
- **remark-gfm**: GitHub Flavored Markdown (хүснэгт, code blocks)
- **rehype-highlight**: Syntax highlighting (highlight.js)
- **Auto-save**: 1 секундын debounce-ээр автоматаар хадгална

### 4.4 Drag-and-Drop (Kanban)

**HTML5 Drag-and-Drop API:**
- `onDragStart`: `dataTransfer.setData('cardId', cardId)`
- `onDragOver`: `e.preventDefault()` (ун altrux зөвшөөрнө)
- `onDrop`: `dataTransfer.getData('cardId')` авч, API руу шинэ баганы ID-гаар шинэчилнэ

## 6. Гүйцэтгэл (Performance)

### 6.1 Frontend гүйцэтгэл

- **Code Splitting**: Next.js автоматаар хуудас бүрийг тусад нь багцалдаг (bundle)
- **Image Optimization**: Next.js Image компонент ашиглан lazy loading
- **Debouncing**: Хуудас бичих үед 1 секундын debounce (автоматаар хадгалах)
- **Virtual Scrolling**: Том жагсаалтын хувьд зөвхөн харагдах хэсгийг л рендэр хийнэ

### 6.2 Backend гүйцэтгэл

- **Database Indexing**: PK, FK, unique талбарууд дээр индекс автоматаар үүснэ
- **Connection Pooling**: Prisma ашиглан PostgreSQL холбоосуудыг pooling хийнэ
- **Query Optimization**: Шаардлагатай талбаруудыг л select хийх

### 6.3 Real-time гүйцэтгэл

- **Event Batching**: Олон үйлдлийг нэгдүгээр илгээх (socket.io acknowledgments)
- **Throttling**: Курсор хөдөлгөх үед 100ms-ийн throttle
- **Room-based**: Зөвхөн холбогдсон өрөөнд л мессеж илгээнэ

## 7. Аюулгүй байдал (Security)

### 7.1 Нэвтрэлтийн аюулгүй байдал

- **JWT Token**: Stateless authentication, HTTP-only cookie
- **bcrypt**: Password hashing (10 salt rounds)
- **HTTPS**: Production орчинд SSL/TLS шифрлэлт

### 7.2 Өгөгдлийн аюулгүй байдал

- **SQL Injection**: Prisma ORM ашиглан автоматаар зайлсан
- **XSS (Cross-Site Scripting)**: Input sanitization, React автоматаар escape хийнэ
- **CSRF**: SameSite cookie attribute ашигласан

### 7.3 Эрх шалгалт (Authorization)

- **Role-based**: OWNER, ADMIN, MEMBER эрхүүд
- **Workspace level**: Зөвхөн гишүүд л хандах эрхтэй
- **Resource level**: Өөрийнх эсэхийг шалгана (жишээ нь: зөвхөн өөрийн мессежийг засах)
