# 2.12 Тестчилэл

## 2.12.1 Тестчилэлийн зорилго

Программ хангамжийн чанарыг баталгаажуулах, системийн үйлдэл бүрийг тодорхой нөхцөлд шалгах зорилготой. Тестчилэл нь дараах зүйлсийг батална:

- Системийн үндсэн функцууд зөв ажиллаж байгаа эсэх
- Алдаатай өгөгдөл боловсруулах чадвар (error handling)
- Хэрэглэгчийн эрх, нэвтрэлт зөв шалгагдаж байгаа эсэх (authentication & authorization)
- Өгөгдлийн бүрэн бүтэн байдал (data integrity)

## 2.12.2 Тестчилэлийн аргачлал

Программд дараах төрлийн тестчилэл хийгдсэн:

### 1. Нэгж тест (Unit Testing)
- Vitest framework ашиглан бичигдсэн
- API endpoint бүрийг тусад нь шалгана
- Express router болон middleware-үүдийг тестэлдэг

### 2. Интеграцын тест (Integration Testing)
- API endpoint-үүд хоорондоо холбогдож ажиллахыг шалгана
- Өгөгдлийн сантай холбогдож бодит мэдээлэл боловсруулахыг тестэлдэг

## 2.12.3 Тестчилсэн модулиуд

### 2.12.3.1 Нэвтрэлт ба бүртгэл (Auth Module)

**Тест хийсэн тохиолдлууд:**

| Тестийн нэр | Тайлбар | Яагаад шалгасан |
|------------|---------|----------------|
| `should register with valid data` | Хүчинтэй мэдээллээр бүртгүүлэх | Системд шинэ хэрэглэгч амжилттай нэмэгдэж байгаа эсэхийг батах |
| `should reject missing email` | Имэйлгүйгээр бүртгүүлэх оролдлого | Заавал шаардлагатай талбарыг шалгах алдаа (validation) |
| `should reject password mismatch` | Нууц үг хоорондоо таарахгүй байх | Хэрэглэгч нууц үгээ зөв баталгаажуулж байгаа эсэх |
| `should reject duplicate email` | Ижил имэйлээр дахин бүртгүүлэх | Өгөгдлийн санд давхардал гарахгүй байдлыг баталгаажуулах |
| `should login with valid credentials` | Зөв имэйл, нууц үгээр нэвтрэх | Нэвтрэлтийн үндсэн функц ажиллаж байгаа эсэх |
| `should reject wrong password` | Буруу нууц үгээр нэвтрэх оролдлого | Эрхгүй хэрэглэгч системд нэвтрэхгүй байдлыг шалгах |
| `should reject non-existent user` | Байхгүй хэрэглэгчээр нэвтрэх | Системд бүртгэлгүй хэрэглэгчийг хүлээн авахгүй байдал |
| `should update username and name` | Профайл мэдээлэл шинэчлэх | Хэрэглэгч өөрийн мэдээллийг өөрчилж чадна эсэх |
| `should reject duplicate username` | Ижил хэрэглэгчийн нэр ашиглах | Хэрэглэгчийн нэр цор ганц байх ёстой тул давхардал шалгах |
| `should reject without auth` | Нэвтрэлтгүйгээр профайл өөрчлөх | Зөвхөн нэвтэрсэн хэрэглэгч мэдээллээ өөрчилж чадна эсэх |
| `should reject invalid token` | Буруу token-оор имэйл баталгаажуулах | Имэйл баталгаажуулалтын аюулгүй байдал |
| `should send reset email for existing user` | Нууц үг сэргээх имэйл илгээх | Нууц үг мартсан тохиолдолд сэргээх боломжтой эсэх |

**Яагаад эдгээр утгаар шалгасан тухай:**
- **Хүчинтэй өгөгдөл**: Системийн хамгийн түгээмэл ашиглагдах тохиолдлыг баталгаажуулах
- **Буруу өгөгдөл**: Хэрэглэгч алдаатай мэдээллээр оролдох үед систем хэрхэн хариу өгөхийг шалгах (error handling)
- **Давхардсан өгөгдөл**: Өгөгдлийн бүрэн бүтэн байдал (data integrity) маш чухал тул давхардал шалгах
- **Нэвтрэлтгүй хандах**: Аюулгүй байдлын хувьд зөвхөн эрхтэй хэрэглэгч л мэдээлэлд хандах ёстой

### 2.12.3.2 Хуудасны модуль (Page Module)

**Тест хийсэн тохиолдлууд:**

| Тестийн нэр | Тайлбар | Яагаад шалгасан |
|------------|---------|----------------|

| `should reject page creation without authentication` | Нэвтрэлтгүйгээр хуудас үүсгэх | Зөвхөн нэвтэрсэн хэрэглэгч хуудас үүсгэх эрхтэй |
| `should reject page creation without workspaceId` | Workspace ID-гүйгээр хуудас үүсгэх | Хуудас ямагт workspace-д хамаарах ёстой тул шалгах |
| `should list pages for a workspace` | Workspace-ийн хуудсуудыг жагсаах | Хуудасны жагсаалт авах функц ажиллаж байгаа эсэх |
| `should retrieve a page by ID` | ID-аар хуудас авах | Тодорхой хуудасны мэдээллийг амжилттай татах |
| `should update a page title and content` | Хуудас засварлах | Хэрэглэгч хуудасны агуулгыг өөрчилж чадна эсэх |

**Яагаад эдгээр утгаар шалгасан тухай:**
- **Workspace-гүй**: Notion-төстэй системд хуудас ямагт workspace-д хамаарна, тул workspaceId заавал шаардлагатай
- **Нэвтрэлтгүй**: Хуудас нь хувийн мэдээлэл тул зөвхөн эрхтэй хэрэглэгч хандах ёстой

## 2.12.4 Тестчилэлийн бүрэн бүтэн байдал

**Backend тестүүд:**

- Нөхөөлсөн модулиуд: auth, chat, drawing, kanban, page, workspace

**Frontend тестүүд:**

**E2E тестүүд:**

- 1 хэрэглэгчийн тохиолдолд collab.spec амжилттай
- 4 spec файл бэлэн

## 2.12.5 Тестчилэлд ашигласан тусгай утгууд

### Яагаад тодорхой утгуудыг сонгосон тухай:

1. **Имэйл хаяг:**
   - `test_' + Date.now() + '@example.com'` - Давхардалгүй, dynamic утга
   - `example.com` - Стандарт test domain, бодит имэйл илгээхгүй

2. **Нууц үг:**
   - `'TestPassword123!'` - Хүчинтэй нууц үгний шаардлагад нийцсэн (том үсэг, жижиг үсэг, тоо, тэмдэгт)
   - `'WrongPassword123!'` - Буруу нууц үгийг шалгахад ашигласан

3. **Урт мэдээлэл:**
   - `'a'.repeat(1001)` - Хязгаараас дээш урт мэдээлэл (1000 тэмдэгт хязгаартай)

4. **Хоосон утга:**
   - `''` - Validation шалгахад ашигласан

5. **Token утгууд:**
   - `'invalid-token'` - Буруу token-ийг шалгах
   - `'Bearer ' + token` - Зөв token формат

## 2.12.6 Тестчилэлийн үр дүн

Тестчилэл хийсний үр дүнд дараах зүйлс баталгаажлаа:

1. **Аюулгүй байдал**: Нэвтрэлт, эрх шалгалт зөв ажиллаж байна
2. **Өгөгдлийн бүрэн бүтэн байдал**: Давхардал, холбоос зөв хадгалагддаг
3. **Алдаа боловсруулах**: Буруу өгөгдөл орж ирэх үед систем сүйрэхгүй, тохирох алдааны мэдээлэл өгдөг
4. **Үндсэн функцууд**: CRUD үйлдлүүд (Create, Read, Update, Delete) бүгд ажиллаж байна

## 2.12.7 Дугтуйл

Тестчилэл нь программын чанарыг баталгаажуулах чухал алхам юм. Одоогийн тестүүд нь системийн үндсэн функцуудыг хамарч, аюулгүй байдал, өгөгдлийн бүрэн бүтэн байдлыг шалгаж чадсан. Гэхдээ илүү өргөн хүрээний тестчилэл (performance, load testing) ирээдүйд хийгдэх шаардлагатай.
